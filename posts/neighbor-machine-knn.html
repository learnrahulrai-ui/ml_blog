The Neighbor Machine: k-Nearest Neighbours, By Hand
Posted: 2026-06-02  |  Author: Rahul Rai  |  Tags: knn, distance, supervised-learning, by-hand
Guess a thing you don't know by copying the things most like it that you DO know.

This is the idea that clicked hardest for me in Lab 1. No dials. No equations to solve. Just: find the places most like this one, copy their prices, average. Let me build the whole machine from nothing.

Two Sides

Split the table into a known pile and a hidden pile.

   LEFT  (known, 80 rows)              RIGHT (hidden, 20 rows)
   rooms school color sun │ price      rooms school color sun │ price
   ──────────────────────────────      ──────────────────────────────
     4     2     6    3   │  3.4          3     2     5    4   │  🔒
     8     9     1    9   │  1.5          ...                  │  🔒
     ...                  │ ...                                │  🔒

The rules of the game:

  • LEFT → read-only. We only LOOK, never change it.
  • RIGHT price → locked. We do NOT peek — that's the truth we score against later.

Distance = How UNLIKE Two Places Are

Distance is built from ALL features at once, squashed into ONE number per left row. 

   hidden right:   rooms 3   school 2   color 5   sun 4

   left row A:     rooms 4   school 2   color 6   sun 3
     gap:            1         0          1        1
                          └──── ONE distance ────┘

   distance = √( gap_rooms² + gap_school² + gap_color² + gap_sun² )

   A: √(1+0+1+1)      = √3   ≈ 1.7   (near)
   B: √(25+49+16+25)  = √115 ≈ 10.7  (far)

Same square → add → root move as the mistake-score. Reused. It's the straight-line distance through feature-space, however many dimensions that space has.

⚠ GOTCHA — distance uses FEATURES only Distance is measured on the features only — never the price. The hidden place has no price, so closeness can't use price. The price only shows up at the very last step, when we average the chosen neighbours.

For One Hidden Row

   1. FIND     ▶ measure distance to ALL 80 left rows   (record each)
   2. PICK     ▶ the 5 SMALLEST distances  (nearest 5)
   3. AVERAGE  ▶ those 5 left PRICES        ▶ one number
   4. WRITE    ▶ append it to this right row ▶ the guess

Each left row does two completely separate jobs:

   [ features ] [ price ]
        │            │
   STEP 1 FIND   STEP 4 AVERAGE
   (who's near)  (only the chosen 5)

The Whole Loop — and the Work It Costs

   for each of the 20 hidden rows:
        sweep all 80 known rows  ─▶  nearest 5  ─▶ average ─▶ guess
   100 rows total  =  80 known  +  20 hidden
   work            =  20 hidden  ×  80 known  =  1600 distance measures

That's the catch with neighbour machines: there's no "training" that compresses anything. Every guess re-scans the entire known pile. It's all memory, no summary. (The textbook calls this a lazy learner.)

The Pencil Version

I did it by hand before typing a line. Five known places, one input (income), a new place at income 5.0, k = 3: 

   Known:  (income, price)
           (5.6, 3.4)   (4.0, 2.7)   (3.8, 2.4)
           (2.5, 1.5)   (7.3, 3.5)

   New place: income = 5.0, k = 3

   Distances from 5.0:
     |5.0 − 5.6| = 0.6  ←  near
     |5.0 − 4.0| = 1.0  ←  near
     |5.0 − 3.8| = 1.2  ←  near
     |5.0 − 2.5| = 2.5       far
     |5.0 − 7.3| = 2.3       far

   3 nearest: 5.6, 4.0, 3.8
   Their prices: 3.4, 2.7, 2.4
   Average = (3.4 + 2.7 + 2.4) / 3 = 2.83

   My guess: $283,000.

The machine does this for 16,000 rows instead of 5. The thinking is identical.

In Code

The ready-made tool hides the loop. fit just memorises the known pile; predict runs the find-pick-average sweep: 

from sklearn.neighbors import KNeighborsRegressor
from sklearn.metrics import mean_squared_error

m = KNeighborsRegressor(n_neighbors=5)
m.fit(X_train, y_train)          # memorize the known pile
guesses = m.predict(X_test)      # find → pick → average for each hidden row

# RMSE: miss → square → mean → root
rmse = mean_squared_error(y_test, guesses) ** 0.5
print(round(rmse, 3))

k Is a Knob

The number of neighbours to ask — k — is a setting I pick before the machine runs. It's not learned. It's chosen. 

   k = 1  →  copy one nearest neighbour (jumpy, fragile)
   k = 25 →  average 25 neighbours      (smooth, maybe too blurry)

Wrap it in a little box: a number goes in, a mistake-score comes out.

def knn_rmse(k):
    m = KNeighborsRegressor(n_neighbors=k)
    m.fit(X_train, y_train)
    guesses = m.predict(X_test)
    return round(mean_squared_error(y_test, guesses) ** 0.5, 3)

Choosing k Fairly

To pick the best k I can't use the sealed test — that's the honesty check. I rotate through the training pile in 5 slices (the cross-validation trick), score each k, and pick the smallest mistake. 

from sklearn.model_selection import KFold, cross_val_score

def cross_val_knn(k_values, X_train, y_train):
    kf = KFold(n_splits=5, shuffle=True, random_state=RANDOM_STATE)
    results = {}
    for k in k_values:
        model = KNeighborsRegressor(n_neighbors=k)
        scores = cross_val_score(model, X_train, y_train, cv=kf,
                                 scoring='neg_root_mean_squared_error')
        results[k] = round(-scores.mean(), 3)
    return results

candidates = [1, 3, 5, 7, 9, 15, 25]
cv = cross_val_knn(candidates, X_train, y_train)
best_k = min(cv, key=cv.get)          # smallest VALUE's key
test_rmse = knn_rmse(best_k)          # open the sealed test ONCE
⚠ GOTCHA — two different "fives" k = how many neighbours to ask (changes the GUESS). 5 slices = how many chunks to grade with (changes the SCORE). They have nothing to do with each other. k=25 with 5 slices is totally fine — different jobs.
⚠ GOTCHA — the negative-RMSE sign flip The scoring tool always chases the biggest number, but mistakes should be small. So it flips the sign: mistakes become negative, and the "biggest" (closest to zero) negative is the smallest mistake. Peel the minus off at the end — that's the -scores.mean() above.
⚠ GOTCHA — smallest KEY vs smallest VALUE min(cv, key=cv.get) returns the key (the k) whose value (the RMSE) is smallest. Plain min(cv) would give the smallest key — wrong. We want the k with the smallest mistake, not the smallest k.

Who Made This (and When)

This "ask the nearest neighbours" rule was written down in 1951 by Evelyn Fix and Joseph Hodges, and sharpened later by Thomas Cover and Peter Hart in 1967. It's older than nearly every programming language you've heard of. So when you do this by hand, you're not just following a recipe — you're re-deriving a 1951 idea on paper.

  1. Two piles: known (read-only) and hidden (price locked).
  2. Distance is one number built from all features — never the price.
  3. For each hidden row: find → pick nearest k → average their prices.
  4. No real training — every guess re-scans the whole known pile.
  5. k is chosen, not learned; pick it by cross-validation, prove on the test once.
