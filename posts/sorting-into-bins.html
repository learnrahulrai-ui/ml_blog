Sorting Into Bins: The S-Curve, the Four-Box Table, and Why Accuracy Lies
Posted: 2026-06-05  |  Author: Rahul Rai  |  Tags: logistic-regression, classification, evaluation, confusion-matrix

Every post so far ended with a number: predicted house price, predicted miles per gallon. A real number on a line. This one ends with a box. A doctor photographs cells from a breast lump and measures 30 things about them — size, texture, smoothness. The question is not "how malignant?" but "malignant or not?" Two bins. One answer. The machine still makes a sliding number internally, but now that number has to be bent, squeezed, and chopped into a bin label.

From Number to Bin

   previous posts (number):   ──────────●──────►   answer = 2.83

   this post (bin):           [ B = well ]  [ M = sick ]   answer = ONE box

The dial-adding machine can only produce a sliding number. To land in one of two bins it needs two more steps: a squash that bends any number into the range 0–1 (so it reads as a chance), and a cutoff that chops that chance into a label. 

   columns ─► ×dials ─► add ─► SQUASH ─► chance 0–1 ─► CUTOFF ─► [B] or [M]

The Squash Curve

The squash curve has an S-shape. Any real number goes in; a chance between 0 and 1 comes out. The formula for the curve is:

σ(z) = 1 / (1 + e−z)

where z is the dial sum: z = β0 + x1β1 + x2β2 + … + x30β30. 

   chance
    1 ┤              ______
  0.5 ┤         ___／            ← dial sum = 0 → chance exactly 0.5
    0 ┤_____／──────────── sum
        −∞     0     +∞

   big negative sum → chance ≈ 0   "well"
   big positive sum → chance ≈ 1   "sick"
   sum = 0          → chance = 0.5  (fence)

At the default cutoff of 0.5, this means: if the sum of weighted measurements tips positive, the lump is called sick; if negative, well. The curve is also doing something deeper — the linear sum z is the log-odds of being sick:

log( P(sick | x) / P(well | x) )  =  β0 + xTβ

Each dial βj is the change in log-odds per one-unit step in its measured column, holding the other 29 fixed. Double a column's value, the log-odds shifts by βj. That's the unit the dials operate in.

Setting the Dials: What the Machine Minimises

In the straight-stick rule the machine minimised squared leftovers (MSE). Here the answer is a label (0 or 1), so squared distance is the wrong ruler. Instead the machine maximises the likelihood of the observed labels, which is equivalent to minimising a cross-entropy loss:

ℒ(β) = −(1/n) Σ [ y log σ(z) + (1−y) log(1−σ(z)) ]

For a lump that is truly sick (y=1): if the machine outputs chance ≈ 1, the first term log(≈1) ≈ 0 — small loss. If it outputs chance ≈ 0 on a sick lump, log(≈0) → −∞ — enormous loss. The loss punishes confidence in the wrong direction. There is no closed-form minimiser here; the dials are found by rolling downhill through the gradient: 

   gradient of 𝓛 with respect to β:   (1/n) Xᵀ(σ(Xβ) − y)

   each step:  β ← β − α · gradient   (α = step size)

In code, the same two-step as before — make an empty machine, show it the study pile: 

log_reg = LogisticRegression(random_state=42)
log_reg.fit(X_train_scaled, y_train)
▸ WHY SCALED INPUTS? The gradient update moves every dial by the same step size multiplied by its column's values. A column measured in thousands (area ≈ 1000) takes dial steps 1000× larger than one measured in hundredths (smoothness ≈ 0.1). The machine lurches around the large column's dial and barely moves the small one's. Put every column on the same ruler first — mean 0, spread 1 — and the step sizes become comparable.

Encoding the Labels

A machine eats numbers, but the sheet holds letters: M (malignant) and B (benign). The translation is a shelf: M→1, B→0. The bin you are hunting gets the 1 — here that is the sick lump, because recall, precision, and every score in this post count the sick detections. 

df['diagnosis'] = df['diagnosis'].map({'M': 1, 'B': 0})
X = df.drop(columns='diagnosis')   # 30 measured columns
y = df['diagnosis']                # 0 or 1
⚠ RUN THE MAP CELL ONLY ONCE If you run the encoding cell a second time, the column already holds 0 and 1. The map searches for 'M' and 'B' — finds nothing — returns all NaN. NaN is a float; every downstream step breaks. Fix: restart the kernel and run top-to-bottom once. There is no shortcut after the NaN has landed.

The Four-Box Table

Every exam lump lands in exactly one of four boxes based on truth and guess: 

               guess SICK(1)    guess WELL(0)
truth SICK(1)  CAUGHT         │ MISSED         ← deadly if large
truth WELL(0)  ALARM          │ CLEAR          ← wasteful if large

rows    = the truth     (sick / well)
columns = the guess     (shouted sick / said well)
each box = a plain count

CAUGHT is the ideal: a sick lump correctly flagged. MISSED is the catastrophe: cancer goes home untreated. ALARM is the waste: a healthy person gets a scare and an unnecessary biopsy. CLEAR is the other good outcome. 

conf_matrix = confusion_matrix(y_test, y_pred)   # truth first, guess second
sns.heatmap(conf_matrix, annot=True, fmt='d')

   grid shape:  [[ CLEAR  ALARM  ]    ← truth well  (row 0)
                 [ MISSED CAUGHT ]]   ← truth sick  (row 1)
⚠ TRUTH FIRST, GUESS SECOND sklearn's confusion_matrix, accuracy_score, recall_score, and precision_score all want (truth, guess) in that order. Swapping them flips rows and columns and misassigns every count to the wrong box.

Four Scores, Four Angles

No single score summarises what a bin-sorter actually does. The four scores each look at a different slice of the four-box table: 

   accuracy  = (CAUGHT + CLEAR) / everyone
   recall    =  CAUGHT / (CAUGHT + MISSED)      ← top row: share of sick caught
   precision =  CAUGHT / (CAUGHT + ALARM)       ← left col: when we cry sick, real sick?
   F1        =  2 · precision · recall / (precision + recall)
ScoreFormula (counts)Punishes
accuracy(TP + TN) / nany wrong answer equally
recallTP / (TP + FN)MISSED (deadly)
precisionTP / (TP + FP)ALARM (wasteful)
F12·P·R / (P+R)either metric being low

In code: 

y_pred = log_reg.predict(X_test_scaled)
accuracy  = accuracy_score(y_test, y_pred)
recall    = recall_score(y_test, y_pred)
precision = precision_score(y_test, y_pred)
f1        = f1_score(y_test, y_pred)

Why Accuracy Alone Lies

The Wisconsin sheet has roughly 63% benign lumps. A machine that calls everything benign — no dials, no squash, just one constant shout of "well" — scores 63% accuracy. It never catches a single sick person. Accuracy counts all boxes equally, so MISSED costs the same as ALARM. But a MISSED sick lump is cancer going home. Those mistakes are not equal. 

   fool machine: predict B for everything
   accuracy  = 0.63   ← looks decent
   recall    = 0.00   ← catches nobody
   precision = N/A    ← never shouts sick, no CAUGHT, undefined
★ IN CANCER SCREENING, RECALL IS THE NORTH STAR Missing a sick person is the catastrophe. You can tolerate some ALARM (extra biopsies cost money, not lives). So the first score to check is recall — how large a share of the sick lumps does the machine catch? Accuracy tells you nothing about which mistake you are making.

F1: Why the Harmonic Mean

F1 is the harmonic mean of precision and recall, not the arithmetic mean. The harmonic mean is dominated by whichever value is smaller. If recall = 0.98 but precision = 0.10, the arithmetic average is 0.54 (looks okay); the harmonic mean F1 = 0.18 (correctly punishes the terrible precision). F1 collapses toward zero when either score is near zero — you cannot paper over one bad score with one great score. 

   F1 = 2 · P · R / (P + R)

   example:  P=0.10,  R=0.98
   arithmetic mean = (0.10 + 0.98) / 2 = 0.54
   harmonic mean   = 2 · 0.10 · 0.98 / (0.10 + 0.98) = 0.18   ← honest
